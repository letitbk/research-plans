description: Retrospectively decompose already-done work into components with full plans — evidence scan, proposed breakdown, bulk-drafted retrospective plans reviewed in one board batch, reconstructed pre-adoption history
argument-hint: [(none = resume/continue) | component name/number to draft next]
allowed-tools: Read, Write, Edit, Glob, Grep, AskUserQuestion, Bash(python3:*), Bash(git:*), Bash(ls:*), Bash(date:*), Bash(mkdir:*)

Adopt a project's already-executed work into the plan-based workflow: turn "5 months of analysis with no plans" into components that each carry a real, honestly-labeled **retrospective** plan. Skill context: `${CLAUDE_PLUGIN_ROOT}/skills/managing-research-plans/SKILL.md`. Mechanics: `${CLAUDE_PLUGIN_ROOT}/skills/managing-research-plans/scripts/results.py` (bundles) and `board.py --gate-batch` (one board session approves the whole batch).

**Requires an initialized project** (`plans/master-plan.md` with its marker). If the repo has substantial history but is not initialized, run `/research-plans:init` first in its **minimal mode** (context + research questions + markers + real-time `Initialized:` stamp; leave the components table to this command) so the component-derivation interview happens exactly once, here. This command never narrates repository history as components — it derives research *activities* (like init) and uses evidence only to know which activities happened, their status, and when. Adopt is backward-looking: it documents work under the CURRENT research direction. To change the direction itself — a new target and new questions over the same repo — use `/research-plans:renew`, which archives the master plan, writes a fresh one, and hands any carried rows that need retrospective plans back to this command (step 1's hand-built-tracker path consumes them as-is).

A retrospective plan is a full 8-section plan carrying `Provenance: retrospective — written <date>; covers work executed <range>`. It is not a lesser plan: it must pass every rubric check. It is honest about postdating the work, cites its evidence in a `Sources` section, and tags anything reconstructed with hindsight `(reconstructed)`. See `/research-plans:review` for how the rubric judges it.

1. **Resolve the mode.** No argument → **resume**: build the worklist from the tracker (rows whose Execution plan cell is `—` and whose status is `done` / `done (verified)` / `in progress`) and continue where a prior pass left off. A component name/number → draft that one next. If the tracker **already enumerates the components** (a hand-built decomposition, e.g. a project someone already restructured): treat the existing rows as the confirmed decomposition — do **not** re-derive or reorder them; skip to step 4, seeding each plan's Sources from that row's Outcome-notes doc citations, and create only the missing execution dirs. `/adopt` never renumbers, reorders, or renames existing rows or dirs (numbers are stable identifiers — see the master-plan doctrine).

2. **Evidence scan** (read-only). Cluster the history into activities: `git log --format='%ad %h %s' --date=short` by period, top-level directories, script and output inventories, and in-repo planning docs (`codex_review_*.md`, `docs/plans/`, `analysis_plan*/`). Then ask the researcher where any out-of-repo plan docs live (e.g. `~/.claude/plans/`) — **never scan outside the repo unprompted** — and read those. Produce an activity digest: each cluster with its date range and the doc(s) that governed it.

3. **Propose the decomposition.** Present candidate components as **research activities serving the research questions** (evidence tells you which happened and their status; the RQs shape how they are named and bounded — "Round 1 collection" is an activity, "wrote fetch.py" is not). Put the proposals inside the AskUserQuestion options, every executed workstream a first-class row (never thematically compressed). Merge/split/rename with the researcher; propose new RQs if the work demands them. Apply `${CLAUDE_PLUGIN_ROOT}/skills/managing-research-plans/references/split-criteria.md` — including its waves rule: a repeated fielding or collection round is its own component, never merged for being small.

4. **Build the tracker and draft plans.** Add the confirmed rows (status from evidence; next-available identity numbers, in reconstructed work order). **Create `plans/execution/<slug>/` for every confirmed row** — including deferred ones — because `results.py stage` and the board both need the dir. Then draft retrospective plans in **batches of ~5**, resumably:
   - Draft each `Provenance: retrospective` `v1.md` from the evidence and sources: fill all eight sections, cite each substantive claim to its dated source/commit, tag hindsight content `(reconstructed)`, and fill the `Sources` table. Snapshot **out-of-repo** source docs into `plans/execution/<slug>/sources/<date>-<name>.md`; **in-repo** docs are cited by path, never copied. Write each as `.draft-v1.md`, with the first-line `<!-- rp-model … -->` provenance marker (prescribed = the `plan` row from `models.py stage plan`, reported = your session model) — see `/research-plans:plan` step 5; it carries unchanged through the batch gate into `v1.md`.
   - **In-progress components:** the retrospective `v1` covers the executed range only and states the boundary date. Resuming the work needs a **prospective `v2`** (`Supersedes: v1`) — ask the researcher, per in-progress component, whether to draft that `v2` now or defer it.
   - **Batch review:** run `python3 <board.py> --gate-batch` once. The researcher reviews all drafted plans in that one session and approves (all / some); each approval writes a durable content-hashed **ticket**, and you then write each approved plan's `vN.md` — the ticket admits it through the sign-off gate. Un-approved drafts stay as drafts for a later pass. A component may legitimately stay `—` (deferred) — never pad a plan. `--gate-batch` refuses to run with fewer than 2 drafts still awaiting approval — a one-component adoption, or resuming an interrupted batch with a single draft left, passes `--allow-single` explicitly.

5. **History and results.** Offer the **history pass**: reconstruct pre-`Initialized` decisions into `plans/history.md` (create it from the template on first use) — date-granularity entries, each citing its evidence, never a fabricated clock time, and **only events before the `Initialized:` cutoff** (present-day decisions belong in the decision log). Then offer `/research-plans:results` (reconcile mode) to backfill retrofit bundles for the done components — after adoption every done component has a *retrospective* plan, so its bundles are `retrofit`, not `planned` (the provenance rule).

6. **Log.** Append one real-time decision-log entry recording the adoption pass — how many components were added, how many plans drafted/approved, the batch id — per the standard format.
