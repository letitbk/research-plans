Capture results for review on the board. Skill context: `${CLAUDE_PLUGIN_ROOT}/skills/managing-planboard/SKILL.md`. Mechanics script: `${CLAUDE_PLUGIN_ROOT}/skills/managing-planboard/scripts/results.py` (python3 only). Requires an initialized project (`plans/master-plan.md` with its marker); if absent, say so and stop.

A bundle is immutable once finalized. It records what the analysis produced (report.md), the exact files (artifacts/, checksum-matched), the code that produced them (scripts/), and the key numbers (manifest metrics). Review happens later, on the board — never here; the bundle's standing state is its validation result.

1. **Resolve the mode.** `$ARGUMENTS` names a component (name or number, via the master plan tracker) → single capture, step 2. For `--adopt`, load `${CLAUDE_PLUGIN_ROOT}/skills/managing-planboard/references/results-adopt.md` and follow **Adopt existing results**. With no argument, load the same reference and follow **Reconcile missing results**.

2. **Reproduce, then gather candidates.** First **regenerate** when the component has a runnable recipe (see **Regeneration and run recipes** in `references/results-adopt.md`): re-running the producing code is how figures/tables/numbers get into the bundle, not scavenging whatever happens to be on disk. Then run `python3 <script> discover` (pass `--dir <repo-relative-path>` for outputs in non-standard locations) and cross-reference: (a) this session's context — what was just produced or regenerated; (b) the component's latest plan `Verification` section — what outputs the plan promised; (c) when session context is thin (mid-session or fresh session), name-and-history heuristics: output filenames sharing tokens with the component slug or number (`02_clean.R` → `02-data-cleaning`), and `git log` on candidate scripts around the component's execution window. Propose; the researcher confirms (interactive mode; autopilot curates without confirming — see step 3). If the component has **no runnable recipe** and nothing on disk, it is a fresh component — report honestly and **stop without creating a bundle** (never pad one); the sole exception is a retrospective report whose figures cannot be reproduced (see **Summary-only bundles** in the reference).

3. **Interview.** **Two modes.** *Interactive capture* (invoked directly, or chained from `/sync`): the interview below runs as written. *Autopilot capture* (chained from the execution loop): SKIP the interview — curation follows **The per-component tail** step 1 in `${CLAUDE_PLUGIN_ROOT}/skills/managing-planboard/references/execution-loop.md` (you author titles/captions/findings, stamp `curatedBy: "agent"`; the researcher's remedy is Reopen → recapture); every rule below about producedBy honesty, renders, and metrics-as-findings applies to both modes. Ask the researcher which artifacts belong in the bundle (multi-select), then for each: title, one-line caption, and the producing script if you cannot identify it from session context. Ask which key numbers to surface as metrics — each metric is a **finding**: `label` + `value` + optional `note`, plus optional `statement` (the claim sentence shown as the finding's headline), `status` (`robust` | `marginal` | `descriptive` | `retracted` | `superseded`), and `artifactIds` (the ids of the artifacts whose figures/tables are embedded directly under this finding on the board). Group each figure/table under the finding it supports. Never guess a producing script — record `producedBy: null` if unknown. **Deliverables follow the project's output conventions** (CLAUDE.md rule 7) when present: prefer the journal-ready renders — a figure's vector PDF/PNG, a table's typeset .png — as the bundle's artifacts. When a finding's only support is a CSV of estimates, offer to render it now (the /journal-tables skill if available, otherwise modelsummary/kableExtra to .png + .tex) before staging. A rendered table artifact records `file` = the .png plus `tex` (the .tex source) and `data` (the estimates CSV) when they exist — copy those files into `artifacts/` too so the board can link them. A standalone CSV or spreadsheet captured without a render is `kind: "other"`.

4. **Stage.** Run `python3 <script> stage --component <NN-slug>` → staging dir. Copy artifacts: `python3 <script> copy --staging <dir> --into artifacts <paths...>`; copy scripts likewise with `--into scripts`. The copy output gives you sha256/bytes/oversized for the manifest.

5. **Write report.md and manifest.json into the staging dir.** report.md is brief and must be self-explanatory to a reader who has not seen the session: open with a one-to-two-line **Data & method** note (dataset and N, method in a phrase, producing script) — enough context to read the figures cold, no more (the board separately shows plan goal and source paths, so do not repeat those). Then: what ran, what came out, how it meets or misses the plan's success criteria, anomalies worth the researcher's eyes; cite artifacts by id. manifest.json fields: `schemaVersion` 1, `component`, `resultsVersion` (finalize renumbers), `planVersion` = the governing plan version, which is the latest canonical vN with a valid signed or amendment trailer (null if none), `provenance` per the **provenance rule** — read the governing plan's `Provenance:` header: a *prospective* plan that predates the work → `"planned"`; a `Provenance: retrospective` plan (the norm after `/planboard:adopt`), or no plan at all → `"retrofit"`, with `planVersion` still citing the plan's vN where one exists so the board links bundle to plan. A retrospective plan documents work it did not prospectively govern, so its bundles are retrofit, never planned — stamping "planned" would be the results-layer analogue of undeclared retrospection, and the value is permanent (bundles are immutable). `trigger` "initial" | "redo-after-review" (when acting on board feedback) | "plan-revision" (first capture after a new plan version), `capturedAt` via `date +"%Y-%m-%d %H:%M"`, `summary`, `metrics` (findings: `label`/`value`/`note?`/`statement?`/`status?`/`artifactIds?` — see step 3), `artifacts` (id/kind/title/caption/file/source/producedBy — a table artifact may also carry `tex` and `data`, staging-relative paths under `artifacts/`, validated at finalize). `source` is `{path, sha256, bytes, oversized}` exactly as the copy output reports it. `producedBy` is `{"script": "scripts/<basename>", "sourcePath": "<repo-relative source path>", "lang": "python"|"R"}` — the `script` path must resolve to a file inside the bundle's `scripts/` dir — plus the run-recipe fields from **Regeneration and run recipes** in `references/results-adopt.md` (`command`/`cwd`/`args`/`expectedOutputs`/`approvedHash`) when the artifact was regenerated or its recipe is known; kind is figure | table | other. **Late captures:** when this session has no evidence of the run that produced the artifacts (backfill of work executed earlier), add `"late": true` and say so in the report — the script snapshots show the code as it is *now*, which may have drifted since the run. The honesty rule is the decision log's: late capture is fine, unlabeled late capture is not.

6. **Validate the bundle (automatic, advisory).** Two independent passes, both advisory — finalize proceeds regardless of either result. **(a) Integrity — every bundle.** `finalize` mechanically computes and seals `manifest.integrity`: artifact copies match their source checksums, every metric references a real artifact, and every *substantive* finding names at least one artifact. A finding is substantive when its `status` is `robust`/`marginal`, or it carries a claim `statement` whose status is not `descriptive`/`retracted`/`superseded`. A `findings-sourced` failure means a substantive finding cites no artifact — the remedy is to attach one or mark the metric `descriptive`; it is surfaced on the board's Output & Validation tab, never blocked. You do not run this pass — finalize does. **(b) Plan-conformance — planned bundles only.** Runs for a bundle whose `provenance` will be `"planned"` and whose component has a governing plan version. Researcher approval does not gate validation eligibility. A canonical amendment is eligible. First run `python3 ${CLAUDE_PLUGIN_ROOT}/skills/managing-planboard/scripts/models.py check` and relay its one-line hint verbatim if it prints one (then proceed with the existing agents — the hint is advisory); if that hint reports the agents were renamed for the planboard rename (restart needed), the `pb-results-validator` is not loaded yet this session, so spawn the anonymous validator below and note the pins reapply after a restart. Spawn ONE independent validator (Task tool, fresh context — it did not do the work): when `.claude/agents/pb-results-validator.md` exists, dispatch it by name (`subagent_type: "pb-results-validator"`) and pass NO model override — the agent file pins model and effort from `plans/model-profile.md` (a request the platform can override: org allowlist, `CLAUDE_CODE_SUBAGENT_MODEL`); otherwise spawn today's anonymous subagent inheriting the session model. Give it: the governing plan path (`plans/execution/<NN-slug>/v<N>.md` from `manifest.planVersion`), the staging dir path (manifest.json draft, report.md, artifacts/, scripts/), `plans/decision-log.md`, and `git log --oneline` for the component's execution window. Require STRICT JSON: `{"steps": [{"planStep": "<each Build step, short name>", "verdict": "followed|amended|deviated-unrecorded|not-executed|unverifiable", "evidence": "<one line citing a file/output/log/commit>"}], "criteria": [{"criterion": "<each success criterion from Goal and success criteria>", "verdict": "met|not-met|partial|unverifiable", "evidence": "<one line>"}], "notes": "<anything the researcher should see>"}`. The `verdict` fields in this JSON are the validation schema's per-check results, not researcher decisions. `amended` means a recorded revision covers the change (a later plan version or a decision-log entry); `deviated-unrecorded` means execution differs and nothing records it. Validate the JSON; repair or re-prompt once on malformed output; still malformed → set `manifest.validation = {"status": "unverifiable", "reason": "invalid validator output"}` and continue. **Derive `status` mechanically — never let the validator pick it:** any `deviated-unrecorded` or `not-executed` step, or any `not-met` criterion → `deviations-found`; else any `amended` step → `conforms-with-amendments`; else all check results resolved → `conforms`; overall evidence insufficient → `unverifiable`. Write `manifest.validation = {status, validatedAt (date +"%Y-%m-%d %H:%M"), planVersion, validator: "subagent", steps, criteria, notes, modelUsage}` into the staged manifest.json — where `modelUsage` is `{"prescribed": <the `results-validation` row from `models.py stage results-validation` as {"model","effort"}, or null>, "reported": {"model": "<the named pb-results-validator agent's pinned model when it ran, else the session model on anonymous fallback>", "effort": <the agent's pinned effort or null>}}` (omit `modelUsage` for not-applicable/skipped) — and render the same content as a readable `validation.md` in the staging dir (status header, a steps table, a criteria table). Echo a `deviations-found` result to the researcher now — the remedy for an unrecorded material deviation is a plan revision via `/planboard:sync`, never a blocked capture. **Skips are recorded, never silent:** retrofit bundle → `{"status": "not-applicable", "reason": "retrofit"}`; Task tool unavailable (headless) or researcher opt-out → `{"status": "skipped", "reason": "<why>"}` — no validation.md in either case. Validation is advisory: finalize proceeds regardless of the validation result. In the execution loop, this validation runs on the STAGED bundle BEFORE finalize and before any tracker or decision-log bookkeeping — a deviations-found result interrupts per the loop's deviation stop instead of merely echoing.

7. **Finalize and confirm on disk.** Run `python3 <script> finalize --staging <dir> --reported-model <the model id you are running as>` (records model provenance — prescribed from the profile's `execute` stage, reported = this capture session; a self-attestation, not confirmed at runtime). On validation failure, fix the staged files and retry. On success, confirm the printed `rN` path exists on disk before reporting. Then **offer the shareable report** (one AskUserQuestion, before anything else): generate `plans/reports/<NN-slug>-r<N>-report.md` for this bundle now? Default to yes when the component's previous bundle already has a report — regeneration continuity. **Skip this offer** when the bundle has no substantive findings — the step-6 rule exactly: no metric is `robust`/`marginal`, and none carries a claim `statement` whose status is not `descriptive`/`retracted`/`superseded`. There is nothing to narrate, `/planboard:report` would refuse (its null-result gate), and the board shows the null-result state instead. On yes, proceed into the full `/planboard:report <NN-slug> r<N>` workflow now (it writes the files and returns; its board-reopen offer does not apply — this flow opens the board itself next). When several bundles were captured in one pass, ask once at the end with a multi-select over all captured bundles instead of per-bundle. Next, append a real-timestamp decision-log entry recording what was captured, any report generated, and why. The log is written **before** the board opens, so a persistent board session never runs ahead of the capture record. Only then open the board on the bundle by proceeding into the full `/planboard:board <NN-slug>:r<N>` workflow — **not** a bare `board.py` launch: the board workflow serves the board **and routes what the researcher submits** (board steps 4–5), including comments and Reopen requests, then logs the exchange; a bare launch would leave that feedback unrouted. When several bundles were captured in one pass (reconcile mode, or chained from `/planboard:sync`), open the board **once** at the end over everything captured for view-only review — never one per bundle, and never a second board if the caller already opened one. Suggest a commit like `plans: results — <NN-slug> r<N> captured` (do not run without approval). **Autopilot mode ends here:** when chained from the execution loop, step 7 is finalize + on-disk verification ONLY — the report offer, the decision-log entry, the board open, and the commit suggestion are all owned by the loop's runbook; skip every one of them and return.
