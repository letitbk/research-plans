Capture results for review on the board. Skill context: `${CLAUDE_PLUGIN_ROOT}/skills/managing-research-plans/SKILL.md`. Mechanics script: `${CLAUDE_PLUGIN_ROOT}/skills/managing-research-plans/scripts/results.py` (python3 only). Requires an initialized project (`plans/master-plan.md` with its marker); if absent, say so and stop.

A bundle is immutable once finalized. It records what the analysis produced (report.md), the exact files (artifacts/, sha256-verified), the code that produced them (scripts/), and the key numbers (manifest metrics). Verdicts happen later, on the board — never here.

1. **Resolve the mode.** `$ARGUMENTS` names a component (name or number, via the master plan tracker) → single capture, step 2. `--adopt` → step 8. **No argument → reconcile mode, step 9.**

2. **Reproduce, then gather candidates.** First **regenerate** when the component has a runnable recipe (see *Regeneration* at the end): re-running the producing code is how figures/tables/numbers get into the bundle, not scavenging whatever happens to be on disk. Then run `python3 <script> discover` (pass `--dir <repo-relative-path>` for outputs in non-standard locations) and cross-reference: (a) this session's context — what was just produced or regenerated; (b) the component's latest plan `Verification` section — what outputs the plan promised; (c) when session context is thin (mid-session or fresh session), name-and-history heuristics: output filenames sharing tokens with the component slug or number (`02_clean.R` → `02-data-cleaning`), and `git log` on candidate scripts around the component's execution window. Propose; the researcher confirms. If the component has **no runnable recipe** and nothing on disk, it is a fresh component — report honestly and **stop without creating a bundle** (never pad one); the sole exception is a retrospective report whose figures cannot be reproduced (see *Regeneration*).

3. **Interview.** Ask the researcher which artifacts belong in the bundle (multi-select), then for each: title, one-line caption, and the producing script if you cannot identify it from session context. Ask which key numbers to surface as metrics — each metric is a **finding**: `label` + `value` + optional `note`, plus optional `statement` (the claim sentence shown as the finding's headline), `status` (`robust` | `marginal` | `descriptive` | `retracted` | `superseded`), and `artifactIds` (the ids of the artifacts whose figures/tables are embedded directly under this finding on the board). Group each figure/table under the finding it supports. Never guess a producing script — record `producedBy: null` if unknown. **Deliverables follow the project's output conventions** (CLAUDE.md rule 7) when present: prefer the journal-ready renders — a figure's vector PDF/PNG, a table's typeset .png — as the bundle's artifacts. When a finding's only support is a CSV of estimates, offer to render it now (the /journal-tables skill if available, otherwise modelsummary/kableExtra to .png + .tex) before staging. A rendered table artifact records `file` = the .png plus `tex` (the .tex source) and `data` (the estimates CSV) when they exist — copy those files into `artifacts/` too so the board can link them. A standalone CSV or spreadsheet captured without a render is `kind: "other"`.

4. **Stage.** Run `python3 <script> stage --component <NN-slug>` → staging dir. Copy artifacts: `python3 <script> copy --staging <dir> --into artifacts <paths...>`; copy scripts likewise with `--into scripts`. The copy output gives you sha256/bytes/oversized for the manifest.

5. **Write report.md and manifest.json into the staging dir.** report.md is brief and must be self-explanatory to a reader who has not seen the session: open with a one-to-two-line **Data & method** note (dataset and N, method in a phrase, producing script) — enough context to read the figures cold, no more (the board separately shows plan goal and source paths, so do not repeat those). Then: what ran, what came out, how it meets or misses the plan's success criteria, anomalies worth the researcher's eyes; cite artifacts by id. manifest.json fields: `schemaVersion` 1, `component`, `resultsVersion` (finalize renumbers), `planVersion` = latest signed vN (null if none), `provenance` per the **provenance rule** — read the governing plan's `Provenance:` header: a *prospective* plan that predates the work → `"planned"`; a `Provenance: retrospective` plan (the norm after `/research-plans:adopt`), or no plan at all → `"retrofit"`, with `planVersion` still citing the plan's vN where one exists so the board links bundle to plan. A retrospective plan documents work it did not prospectively govern, so its bundles are retrofit, never planned — stamping "planned" would be the results-layer analogue of undeclared retrospection, and the value is permanent (bundles are immutable). `trigger` "initial" | "redo-after-review" (when acting on board feedback) | "plan-revision" (first capture after a new plan version), `capturedAt` via `date +"%Y-%m-%d %H:%M"`, `summary`, `metrics` (findings: `label`/`value`/`note?`/`statement?`/`status?`/`artifactIds?` — see step 3), `artifacts` (id/kind/title/caption/file/source/producedBy — a table artifact may also carry `tex` and `data`, staging-relative paths under `artifacts/`, validated at finalize). `source` is `{path, sha256, bytes, oversized}` exactly as the copy output reports it. `producedBy` is `{"script": "scripts/<basename>", "sourcePath": "<repo-relative source path>", "lang": "python"|"R"}` — the `script` path must resolve to a file inside the bundle's `scripts/` dir — plus the run-recipe fields from *Regeneration* (`command`/`cwd`/`args`/`expectedOutputs`/`approvedHash`) when the artifact was regenerated or its recipe is known; kind is figure | table | other. **Late captures:** when this session has no evidence of the run that produced the artifacts (backfill of work executed earlier), add `"late": true` and say so in the report — the script snapshots show the code as it is *now*, which may have drifted since the run. The honesty rule is the decision log's: late capture is fine, unlabeled late capture is not.

6. **Validate execution against the plan (automatic, advisory).** Runs for a bundle whose `provenance` will be `"planned"` and whose component has a signed plan. First run `python3 ${CLAUDE_PLUGIN_ROOT}/skills/managing-research-plans/scripts/models.py check` and relay its one-line hint verbatim if it prints one (then proceed with the existing agents — the hint is advisory). Spawn ONE independent validator (Task tool, fresh context — it did not do the work): when `.claude/agents/rp-results-validator.md` exists, dispatch it by name (`subagent_type: "rp-results-validator"`) and pass NO model override — the agent file pins model and effort from `plans/model-profile.md` (a request the platform can override: org allowlist, `CLAUDE_CODE_SUBAGENT_MODEL`); otherwise spawn today's anonymous subagent inheriting the session model. Give it: the signed plan path (`plans/execution/<NN-slug>/v<N>.md`), the staging dir path (manifest.json draft, report.md, artifacts/, scripts/), `plans/decision-log.md`, and `git log --oneline` for the component's execution window. Require STRICT JSON: `{"steps": [{"planStep": "<each Build step, short name>", "verdict": "followed|amended|deviated-unrecorded|not-executed|unverifiable", "evidence": "<one line citing a file/output/log/commit>"}], "criteria": [{"criterion": "<each success criterion from Goal and success criteria>", "verdict": "met|not-met|partial|unverifiable", "evidence": "<one line>"}], "notes": "<anything the researcher should see>"}`. `amended` means a recorded revision covers the change (a later plan version or a decision-log entry); `deviated-unrecorded` means execution differs and nothing records it. Validate the JSON; repair or re-prompt once on malformed output; still malformed → set `manifest.validation = {"status": "unverifiable", "reason": "invalid validator output"}` and continue. **Derive `status` mechanically — never let the validator pick it:** any `deviated-unrecorded` or `not-executed` step, or any `not-met` criterion → `deviations-found`; else any `amended` step → `conforms-with-amendments`; else all verdicts resolved → `conforms`; overall evidence insufficient → `unverifiable`. Write `manifest.validation = {status, validatedAt (date +"%Y-%m-%d %H:%M"), planVersion, validator: "subagent", steps, criteria, notes}` into the staged manifest.json, and render the same content as a readable `validation.md` in the staging dir (status header, a steps table, a criteria table). Echo a `deviations-found` result to the researcher now — the remedy for an unrecorded material deviation is a plan revision via `/research-plans:sync`, never a blocked capture. **Skips are recorded, never silent:** retrofit bundle → `{"status": "not-applicable", "reason": "retrofit"}`; Task tool unavailable (headless) or researcher opt-out → `{"status": "skipped", "reason": "<why>"}` — no validation.md in either case. Validation is advisory: finalize proceeds regardless of the verdict.

7. **Finalize and verify on disk.** Run `python3 <script> finalize --staging <dir>`. On validation failure, fix the staged files and retry. On success, verify the printed `rN` path exists on disk before reporting. Then **offer the shareable report** (one AskUserQuestion, before anything else): generate `plans/reports/<NN-slug>-r<N>-report.md` for this bundle now? Default to yes when the component's previous bundle already has a report — regeneration continuity. On yes, proceed into the full `/research-plans:report <NN-slug> r<N>` workflow now (it writes the files and returns; its board-reopen offer does not apply — this flow opens the board itself next). When several bundles were captured in one pass, ask once at the end with a multi-select over all captured bundles instead of per-bundle. Next, append the step-10 decision-log entry now — the log is written **before** the board opens, so a persistent board session never runs ahead of the capture record. Only then open the board on the bundle by proceeding into the full `/research-plans:board <NN-slug>:r<N>` workflow — **not** a bare `board.py` launch: the board workflow serves the board **and routes what the researcher submits** (board steps 4–5), which is how a verdict actually gets recorded (`verdict.json` via `results.py verdict`), the tracker updated, and the exchange logged; a bare launch would leave the verdict unapplied. When several bundles were captured in one pass (reconcile mode, or chained from `/research-plans:sync`), open the board **once** at the end over everything captured — never one per bundle, and never a second board if the caller already opened one. Suggest a commit like `plans: results — <NN-slug> r<N> captured` (do not run without approval).

8. **Adopt mode (--adopt).** For pre-existing figures/tables made before or outside any plan. Run discover, present the candidates grouped by directory, and interview: which artifacts matter, and which component each belongs to — offer to add a tracker row for work that has no component yet (status from evidence, notes say "retrofit"). Then per component follow steps 4-7 with `provenance` "retrofit" and `planVersion` = latest signed version or null (validation records `not-applicable` per step 6). Retrofit bundles review and verdict identically; the provenance chip keeps the record honest.

9. **Reconcile mode (no argument).** Component-first backfill for a project whose plans ran ahead of its results record. Build the worklist from the tracker and disk:
   - components `done` or `done (verified)` or `in progress` whose `plans/execution/<NN-slug>/results/` has no `r*/` bundle;
   - components whose latest bundle has drifted sources (`python3 <script> changed --component <NN-slug>`);
   - leftover `results/.staging-*` dirs (interrupted captures — offer to resume or discard).

   Present the worklist and let the researcher choose: walk all, pick some, or skip. Then per component, one at a time in tracker order, run steps 2-7 — applying the **provenance rule** (step 5) to each bundle: a *prospective* signed plan that predates the work → `provenance` "planned" + `late: true` (backfill); a `Provenance: retrospective` plan, or no plan at all → `provenance` "retrofit" (`planVersion` still citing the retrospective vN where one exists). After `/research-plans:adopt`, every done component has a *retrospective* plan, so its backfilled bundles are **retrofit**, not planned — the plan links them (`planVersion`) without claiming to have governed them. A component with nothing qualifying gets reported as such (and, if the researcher agrees, a one-line tracker note); the only zero-artifact bundle ever finalized is a retrospective report whose figures cannot be reproduced, and only with the researcher's explicit confirmation (see *Regeneration*). Finish by opening one board session for a verdict pass over everything captured. Never capture all components silently in bulk: the per-component interview is the verification; skipping it is how plan theater starts.

10. **Log.** The decision-log entry (real timestamp) recording what was captured, any report generated, and why, per the standard format — written in step 7 **before** the board opened; this step is only a checkpoint that it happened.

## Regeneration, the run recipe, and summary-only bundles

**Reproducibility-first capture.** When a component has runnable producing code, reproduce its outputs rather than scavenging stale files. Identify the recipe from the plan's `Verification`/build steps, this session's context, or the latest bundle's `producedBy` on a re-capture; ask the researcher if unclear. A recipe is a producing script plus how to run it:

- `command` — inferred from the script extension: `.R` → `Rscript`, `.py` → `python3` (fall back to `python`), `.sh` → `bash`. Unsupported interpreters (e.g. Stata `.do`) have no recipe → ask the researcher for the command or stop; never treat as "no code."
- `cwd` (repo-relative, default repo root), `args`, and `expectedOutputs` (repo-relative files the run should produce).
- `approvedHash` — the sha256 of the script source at approval; the trust gate for no-prompt auto-run.

**Running.** Resolve the script and confirm it is repo-relative and inside the repo (reject `..` and symlink escapes). Auto-run **without asking** only when the script's current sha256 matches `approvedHash`; otherwise show the exact command and ask first. Record a run-start timestamp, then run from `cwd` with `set -o pipefail`, teeing stdout+stderr to `logs/$(date +%Y-%m-%d_%H-%M-%S)_results-regenerate-<slug>.log`. **Stop the capture** — build no bundle — if the run exits non-zero, or if any `expectedOutput` is missing or was not created/modified after run-start; cite the log. Run only this component's recipes, never unrelated code. A capture that regenerated this session is not `late`; a retrospective one that reused old files (or is report-only) stays `late: true`.

**Summary-only (report-only) bundles.** A bundle with zero artifacts is finalized in exactly one situation: a **retrospective / `--adopt` / reconcile** capture where the researcher has a report to record but the figures cannot be reproduced (no runnable recipe, or the sources are gone) — and only after the researcher **explicitly confirms**. An initial/planned capture of a fresh component with nothing to reproduce **stops without a bundle** (step 2); the board shows its top-level empty state. The board renders a "summary only" notice on the report-only bundle so its empty gallery reads as expected rather than broken.
