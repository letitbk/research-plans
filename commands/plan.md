Author an execution plan for one component, jointly with the researcher. Skill context: `${CLAUDE_PLUGIN_ROOT}/skills/managing-research-plans/SKILL.md`. This command requires an initialized project (`plans/master-plan.md` with its marker); if absent, say so and stop — suggest `/research-plans:init`.

**Model nudge** (before step 1). Run `python3 ${CLAUDE_PLUGIN_ROOT}/skills/managing-research-plans/scripts/models.py stage plan`. Empty output → no profile or no usable row → say nothing and continue; relay any stderr warning once (a malformed profile row is fixed by `/research-plans:models`). Output is a JSON row; when its model is not `inherit` and differs from the model you are running as (you know your own identity), print exactly one line — `Model profile: this stage is set to <model>; you're on <current>. Switch with /model <model> (safe mid-conversation — nothing is lost), or continue as-is.` — substituting the profile's model and your current one, appending `, effort <level>` to the `/model` suggestion when the row names an effort and your build exposes a session effort control. Then proceed; never block on the nudge, never repeat it later in the session.

1. **Pick the component.** Read `plans/master-plan.md`, including its `### Research questions` list and the component's `Serves` value — the plan you are about to author is anchored to those questions. If `$ARGUMENTS` names or numbers a component, use it; otherwise propose the first `not started` row (table order is the default sequence) and confirm.

   **Mid-session intake.** If planning follows same-session exploratory work (including mid-session adoption), harvest it: choices already made and their observed reasons become grounded Whys in Scope decisions and content for the Goal section. Do not backfill decision-log entries for pre-plan exploration — the log records the planning dialogue from now on.

2. **Scope check before authoring.** Judge the component against `${CLAUDE_PLUGIN_ROOT}/skills/managing-research-plans/references/split-criteria.md`. If it spans independent parts, propose the split first: new named components as tracker rows, then plan only the one being worked on next.

3. **Offer bounded exploration.** If the component is data-facing and the data has not been explored, offer a short, read-only exploration per `${CLAUDE_PLUGIN_ROOT}/skills/managing-research-plans/references/explore-before-planning.md` before authoring. Findings feed the Scope decisions table; surprises go to the decision log.

4. **Co-author through dialogue, not generation.** The FIRST dimension is the goal: what this component will produce or establish, which research question(s) it serves (confirm or correct the tracker's Serves value), and the concrete success criteria the researcher will judge it by — these become the `## Goal and success criteria` section with its `Serves:` line, and every build step must trace back to them. Then work dimension by dimension through the substantive choices (sample, measures, model or method, robustness, outputs — the deliverable figures/tables per CLAUDE.md rule 7's output conventions — whatever fits the component), using AskUserQuestion. For each consequential choice:
   - present the live options with trade-offs,
   - **require a reason with depth, tied to the goal** — the Why is the point of the artifact, and the rubric's spine scores whether it goes beyond restating the choice and connects to the research question. The researcher authoring the reason, choosing among options, or approving a proposed one all count fully — what matters is that a grounded reason lands on the page, not who first phrased it.
   - **push back on a bare pick on a consequential fork**: if the researcher waves a high-stakes choice through, say what the default assumes and ask if that assumption holds here. Draw the researcher's knowledge out; do not substitute your own.
   - if you must settle a consequential call yourself (the researcher isn't available for it), **flag it in the decision log** as an agent call for the researcher to confirm — never settle it silently.
   - Append decision-log entries **as the dialogue happens** (real timestamps via `date +"%Y-%m-%d %H:%M"`), not afterward.

5. **Write the plan.** Determine `NN` (next number in `plans/execution/`) and a short slug. **Overwrite guard:** the target is `plans/execution/<NN-slug>/v1.md`; if that file already exists, stop and ask — never overwrite any `vN.md`. Fill the template at `${CLAUDE_PLUGIN_ROOT}/skills/managing-research-plans/templates/execution-plan.md`: one continuous narrative (Context first, then Goal and success criteria, Decisions, Approach, Build steps, Verification, Out of scope; Files to reuse only if real), no Supersedes line for v1, the researcher's choices and reasons in the Decisions table. **Write it short and in plain language** — a plan is read cold, so carry only what the rubric scores and cut the rest; push code, exact commands, and dense technical detail into inline `<details class="agent-detail">` blocks so the default read stays scannable. Make Verification a real test that the success criteria were met (validation), and Out of scope name both what's excluded and what not to touch. What goes in: the choices and their reasons. What stays out: speculation about results, agent boilerplate, padding.

   **Stamp model provenance (first line).** Make the very first line of the plan file — above the `#` title — a single marker: `<!-- rp-model {"prescribed":P,"reported":{"model":"<the model id you are running as>","effort":null}} -->`, where `P` is the `plan` row from the model nudge above as `{"model":...,"effort":...}` (or `null` if there was no profile row). Write this line into `.draft-v1.md` from the start and carry it **unchanged** into every `v1-draft-K.md` snapshot and the final `v1.md` — the sign-off gate must hash the same bytes it approves, so it cannot be added after approval. The board strips this line before rendering and shows it as a model chip.

6. **Sign-off.** Read the finished draft back in brief. A board preview is still available during the dialogue (write the draft to `plans/execution/<NN-slug>/.draft-v1.md` and run `/research-plans:board <NN-slug>`; fold annotations into the draft), but **formal approval happens at the sign-off gate**: writing `v1.md` automatically opens the plan in the researcher's browser, where they approve or request changes. **Preserve the drafting trail:** each time you revise the draft for a fresh review round, first snapshot the *current* `.draft-v1.md` to the next `plans/execution/<NN-slug>/v1-draft-<K>.md` (K from 1, the next unused number — never overwrite a snapshot). These commit as the plan's iteration history and are read-only on the board; the sign-off gate ignores them (only `vN.md` is gated). Before writing `v1.md`, make sure any preview board session has been submitted or closed — the gate needs the board. Add the sign-off line with the researcher's name and today's date, then write `v1.md`. If the gate denies, read `plans/.board-feedback.md`, revise the draft to address all feedback, and write again (the gate reopens). If the gate **times out** instead (the researcher wasn't at the browser), the proposal is saved back as the component's `.draft-vN.md`: relaunch `/research-plans:board <NN-slug>` and tell the researcher the draft awaits their Approve there — approving mints a durable ticket and the board workflow's routing performs the ticketed write; never bypass the gate and never use `--gate-batch` for this. After approval lands: delete only the ephemeral `.draft-v1.md` (the committed `v1-draft-*.md` snapshots stay as history), update the component's tracker row to `planned` with a link to the plan, and update `Last updated:`.

7. **Suggest a commit** such as `plan: <NN-slug> v1` (do not run without approval).

8. **Continue the loop automatically.** After a clean sign-off (the gate approved `v1.md` and its browser board has closed), proceed without waiting to be re-invoked: first run the `/research-plans:review` workflow on the signed `v1.md` and save its scorecard, so the researcher immediately sees how the plan scores; then open the board **once** by proceeding into the full `/research-plans:board <NN-slug>` workflow — **not** a bare `board.py` launch: the board workflow's serve-and-route steps (board steps 4–5) are what record anything the researcher submits (discuss comments, draft revisions, log the exchange), so a bare launch would strand their annotations in `plans/.board-feedback.md`. Never run two boards at once — the sign-off gate's own board must have closed first (`.board.lock` rejects a concurrent board). If the gate denied and you are still revising the draft, do **not** start the review; the chain begins only after `v1.md` is signed.
