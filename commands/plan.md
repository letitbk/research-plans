Author execution plans for one or more components, jointly with the researcher. Skill context: `${CLAUDE_PLUGIN_ROOT}/skills/managing-research-plans/SKILL.md`. Load `${CLAUDE_PLUGIN_ROOT}/skills/managing-research-plans/references/planning-doctrine.md` now — it is the authoring standard steps 3–5 follow. This command requires an initialized project (`plans/master-plan.md` with its marker); if absent, say so and stop — suggest `/research-plans:init`.

**Model nudge** (before step 1). Run `python3 ${CLAUDE_PLUGIN_ROOT}/skills/managing-research-plans/scripts/models.py stage plan`. Empty output → no profile or no usable row → say nothing and continue; relay any stderr warning once (a malformed profile row is fixed by `/research-plans:models`). Output is a JSON row; when its model is not `inherit` and differs from the model you are running as (you know your own identity), print exactly one line — `Model profile: this stage is set to <model>; you're on <current>. Switch with /model <model> (safe mid-conversation — nothing is lost), or continue as-is.` — substituting the profile's model and your current one, appending `, effort <level>` to the `/model` suggestion when the row names an effort and your build exposes a session effort control. Then proceed; never block on the nudge, never repeat it later in the session.

1. **Pick the component or components.** Read `plans/master-plan.md`, including its `### Research questions` list and each component's `Serves` value — every plan is anchored to those questions. `$ARGUMENTS` may name or number one or more components. Resolve the whole set before any planning dialogue: an unknown name or number is an error that names the valid tracker rows and stops; name and skip a component that already has a signed plan; resume an existing unsigned draft rather than creating a duplicate. With no arguments, propose the first `not started` row (table order is the default sequence) and confirm. For multiple components, establish shared project and research-question context once, then co-author each plan sequentially through steps 2–5. Each component ends with a queued `.draft-v1.md` and a freshly scored draft scorecard; do not launch a sign session until the full requested set has been drafted.

   **Mid-session intake.** If planning follows same-session exploratory work (including mid-session adoption), harvest it: choices already made and their observed reasons become grounded Whys in Scope decisions and content for the Goal section. Do not backfill decision-log entries for pre-plan exploration — the log records the planning dialogue from now on.

2. **Scope check before authoring.** Judge the component against `${CLAUDE_PLUGIN_ROOT}/skills/managing-research-plans/references/split-criteria.md`. If it spans independent parts, propose the split first: new named components as tracker rows, then plan only the one being worked on next.

3. **Ground before authoring (read-only, default).** Follow the doctrine's research-first pass: repo structure, data presence and rough shape, prior components' outputs, existing scripts in this component's area — bounded (roughly a dozen files, a few read-only commands), findings summarized in 2–3 sentences before the first question. The researcher can decline with "skip exploration". If the component is data-facing and the data has not been explored, offer the deeper pass per `${CLAUDE_PLUGIN_ROOT}/skills/managing-research-plans/references/explore-before-planning.md`. Findings feed the Scope decisions table; surprises go to the decision log.

4. **Co-author through dialogue, not generation.** The FIRST dimension is the goal: what this component will produce or establish, which research question(s) it serves (confirm or correct the tracker's Serves value), and the concrete success criteria the researcher will judge it by — these become the `## Goal and success criteria` section with its `Serves:` line, and every build step must trace back to them. Then work dimension by dimension through the substantive choices (sample, measures, model or method, robustness, outputs — the deliverable figures/tables per CLAUDE.md rule 7's output conventions — whatever fits the component), using AskUserQuestion. For each consequential choice:
   - present the live options with trade-offs and the assumption each default rests on,
   - **require a reason with depth, tied to the goal** — the Why is the point of the artifact, and the rubric's spine scores whether it goes beyond restating the choice and connects to the research question. The researcher authoring the reason, choosing among options, or approving a proposed one all count fully — what matters is that a grounded reason lands on the page, not who first phrased it.
   - **push back on a bare pick on a consequential fork**: if the researcher waves a high-stakes choice through, say what the default assumes and ask if that assumption holds here. Draw the researcher's knowledge out; do not substitute your own.
   - if you must settle a consequential call yourself (the researcher isn't available for it), **flag it in the decision log** as an agent call for the researcher to confirm — never settle it silently.
   - Append decision-log entries **as the dialogue happens** (real timestamps via `date +"%Y-%m-%d %H:%M"`), not afterward.

5. **Write the plan.** Determine `NN` (next number in `plans/execution/`) and a short slug. **Overwrite guard:** the target is `plans/execution/<NN-slug>/v1.md`; if that file already exists, stop and ask — never overwrite any `vN.md`. Fill the template at `${CLAUDE_PLUGIN_ROOT}/skills/managing-research-plans/templates/execution-plan.md`: one continuous narrative (Context first, then Goal and success criteria, Decisions, Approach, Build steps, Verification, Out of scope; Files to reuse only if real), no Supersedes line for v1, the researcher's choices and reasons in the Decisions table. **Write it short and in plain language** — a plan is read cold, so carry only what the rubric scores and cut the rest; **bold each Decisions row's keyword and each build step's verb phrase, use *italics* for rationale asides, keep paragraphs under four lines, and give each build step one sentence** (elaboration in an indented follow-up line — the board renders steps as numbered cards); push code, exact commands, and dense technical detail into inline `<details class="agent-detail">` blocks so the default read stays scannable. Make Verification a real test that the success criteria were met (validation), and Out of scope name both what's excluded and what not to touch. What goes in: the choices and their reasons. What stays out: speculation about results, agent boilerplate, padding.

   **Stamp model provenance (first line).** Make the very first line of the plan file — above the `#` title — a single marker: `<!-- rp-model {"prescribed":P,"reported":{"model":"<the model id you are running as>","effort":null}} -->`, where `P` is the `plan` row from the model nudge above as `{"model":...,"effort":...}` (or `null` if there was no profile row). Write this line into `.draft-v1.md` from the start and carry it **unchanged** into every `v1-draft-K.md` snapshot and the final `v1.md` — the sign-off gate must hash the same bytes it approves, so it cannot be added after approval. The board strips this line before rendering and shows it as a model chip.

6. **Prepare the pending plan.** Read the finished draft back in brief. Write it to `plans/execution/<NN-slug>/.draft-v1.md` without a trailer. Keep the model marker from step 5 as line 1. Run the `/research-plans:review` workflow on the draft so the board can show a current scorecard. Update the tracker row to `planned` and link it to the draft path. Here, `planned` means the component has a draft or signed plan, so `/research-plans:execute` can still propose it.

   Say `draft ready — it signs at /research-plans:execute, or run /research-plans:sign <component> to commit it now`. The persistent board is optional for reading, annotations, diffs, and extra reviews. It has no approval action. If the researcher requests changes, snapshot the current draft to the next unused `v1-draft-<K>.md`, revise the draft, and run a fresh review. If they want to sign now, follow **Launching a sign session** in `${CLAUDE_PLUGIN_ROOT}/skills/managing-research-plans/references/sign-off.md`. For two or more drafts, launch one sign session for the full set. Follow **Recovery** in that reference after any interruption. Finalize approved items with **The finalization transaction**, and keep their tracker status `planned`.

7. **Commit rides the execute prompt.** Do not suggest a separate plan commit here. The execute prompt asks for commit consent after the pending plan has been signed.

8. **Offer execution.** Proceed into `/research-plans:execute` for the prepared component or set. That command signs pending drafts before it follows **The execute prompt** in `${CLAUDE_PLUGIN_ROOT}/skills/managing-research-plans/references/execution-loop.md`. On `later`, say `/research-plans:execute <component>` resumes from the pending draft and stop.
