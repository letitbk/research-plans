Score one execution plan against the rubric at `${CLAUDE_PLUGIN_ROOT}/skills/managing-research-plans/references/plan-rubric.md` (v0.4, five channels, 0–3 each). There is no pass/fail threshold — a plan is scored, not gated. Requires an initialized project; if `plans/master-plan.md` is absent, say so and stop.

1. **Resolve the target.** `$ARGUMENTS` may be a path or a component name/number; default is the latest version of the `in progress` (else most recently `planned`) component. Read the plan and the master plan (research questions, Serves values). Gather git evidence for the non-scored **integrity flags** only (never for the channel scores): `git log --follow --format='%ad %s' -- <plan path>` for commit timing, the component directory for version history, and — for a `Provenance: retrospective` plan — its cited Sources. The flags: `uncommitted` (the plan was not committed before its governed work), `unrecorded-deviation` (execution departed the plan with no recorded amendment), `unsupported-sources` (a retrospective plan whose Sources do not resolve to dated evidence in the coverage range). Flags are reported beside the score; they never change a channel score.

2. **Dispatch or run inline.** When `.claude/agents/rp-plan-reviewer.md` exists, the scoring runs in ONE dispatched `rp-plan-reviewer` subagent — its model and effort come from `plans/model-profile.md`, a request the platform can override (org allowlist, `CLAUDE_CODE_SUBAGENT_MODEL`); when it does not exist, run step 3 inline exactly as written (no anonymous-subagent fallback). Before dispatching, run `python3 ${CLAUDE_PLUGIN_ROOT}/skills/managing-research-plans/scripts/models.py check` and relay its one-line hint verbatim if it prints one (advisory — proceed with the existing agent). Dispatch one Task with `subagent_type: "rp-plan-reviewer"`, NO model override, giving it: the plan path, the master plan path, `plans/decision-log.md`, the component's execution directory, the rubric path, the split-criteria path (`${CLAUDE_PLUGIN_ROOT}/skills/managing-research-plans/references/split-criteria.md`), the rubric version to stamp (`0.4`), today's date (`date +%Y-%m-%d`), and the step-1 resolution. The agent returns ONLY the scorecard JSON (the `board-scorecard` fence shape, schemaVersion 3). Validate it: parseable JSON; `status` ∈ `scored`/`unscorable`; a `scored` card has exactly the five channels (`goal`, `decisions`, `steps`, `validation`, `boundaries`) with integer `score` 0–3 and a `total` equal to their sum. Repair or re-prompt once on malformed output; still malformed → say so and run step 3 inline.

3. **Score inline (skipped when step 2 dispatched).** Apply the rubric exactly. First the **readability precondition**: if the five channels' content cannot be extracted from the plan at all, report `Status: unscorable — <what is missing>` and STOP (do not fabricate scores). Otherwise score each of the five channels 0–3 with a short quoted evidence span, following the anchors. Score **from the plan itself** — not the chat or decision log. For channel 2, credit a choice among options, an approval, or joint authorship as full control; score the depth of the reasoning and its link to the research goal, not who phrased it. Then report: the **profile** (`G · D · S · V · B = <total>/15`), the **biggest leak** (the lowest channel, "where the most authorship is being handed to the agent"), the **unresolved forks** (the open decisions dragging the score down — the fix list for the next revision), **one suggested move per leak**, the non-scored **integrity flags** from step 1, and the **split assessment** per `${CLAUDE_PLUGIN_ROOT}/skills/managing-research-plans/references/split-criteria.md`. Read the profile, not the total: name any channel scoring 0 as a missing control channel.

4. **Save the scorecard** to `plans/reviews/<NN-slug>-v<N>.md` using the template at `${CLAUDE_PLUGIN_ROOT}/skills/managing-research-plans/templates/review-scorecard.md` (schemaVersion 3; prose and JSON must agree — the board renders the JSON). Add a `"modelUsage"` field: `{"prescribed": <the `plan-review` row from `models.py stage plan-review` as {"model","effort"}, or null>, "reported": {"model": "<the dispatched agent's pinned model, or the session model inline>", "effort": <the pinned effort, or null inline>}}` — the board shows it as a "reviewed by" chip. **Idempotent, non-blocking save:** when this command is invoked automatically after a sign-off (by `/plan`, `/sync`, `/adopt`, or a board Approve), save without prompting; if a valid v3 scorecard already exists for this exact `planPath`, skip re-scoring; if scoring failed, write nothing and report "score unavailable" — never block the sign-off. When invoked interactively and the file exists, ask before overwriting. The score appears in the plan's header on the board (`/research-plans:board`).

5. **If the researcher accepts a revision or a split, that is a decision** — append it to `plans/decision-log.md` (real timestamp), and route the change properly: plan revisions go through a new signed version (see `/research-plans:sync` step 6 semantics — never edit `vN.md`), splits go through tracker rows.
