description: Review an execution plan in two stages — threshold verdict (is this a plan?) then quality grade — with a split assessment
argument-hint: [plan path or component name/number]
allowed-tools: Read, Write, Edit, Glob, Grep, AskUserQuestion, Task, Bash(python3:*), Bash(git:*), Bash(ls:*), Bash(date:*), Bash(mkdir:*)

Review one execution plan against the rubric at `${CLAUDE_PLUGIN_ROOT}/skills/managing-research-plans/references/plan-rubric.md` (v0.3, two-stage, provenance-aware). Requires an initialized project; if `plans/master-plan.md` is absent, say so and stop.

1. **Resolve the target.** `$ARGUMENTS` may be a path or a component name/number; default is the latest version of the `in progress` (else most recently `planned`) component. Read the plan (including its `Provenance:` header — absent = prospective); read the master plan (research questions, Serves values, and the `Initialized:` line). Gather git evidence for T8/T9: `git log --follow --format='%ad %s' -- <plan path>` for commit timing, the component directory for version history, and `plans/decision-log.md` for engagement traces. **Adoption cutoff:** evidence before the master plan's `Initialized:` timestamp (fall back to the git first-commit of master-plan.md when the line is absent) does not count against T8/T9 — no plan governed that period. **Cutoff × declaration:** this exemption applies only when the plan does not *claim* to cover pre-`Initialized` work; a plan whose governed work predates `Initialized:` but omits `Provenance: retrospective` is undeclared retrospection (T8 `fail`), not exempt. **For a `Provenance: retrospective` plan, resolve its Sources before scoring T8:** open each cited in-repo doc and check its `git log` dates fall within or before the declared coverage range; for snapshotted out-of-repo sources, confirm the snapshot exists under `sources/` and, when a batch sign-off ticket recorded source hashes, that they match. Unresolvable or undated Sources → T8 `unknown`; a cited source that postdates its work, or a declaration with no resolvable Sources → T8 `fail`.

2. **Dispatch or run inline.** When `.claude/agents/rp-plan-reviewer.md` exists, both scoring stages run in ONE dispatched `rp-plan-reviewer` subagent — its model and effort come from `plans/model-profile.md`, a request the platform can override (org allowlist, `CLAUDE_CODE_SUBAGENT_MODEL`); when it does not exist, run steps 3–4 inline exactly as written (no anonymous-subagent fallback for this command). Before dispatching, run `python3 ${CLAUDE_PLUGIN_ROOT}/skills/managing-research-plans/scripts/models.py check` and relay its one-line hint verbatim if it prints one (advisory — proceed with the existing agent). Dispatch one Task with `subagent_type: "rp-plan-reviewer"`, NO model override, giving it: the plan path, the master plan path, `plans/decision-log.md`, the component's execution directory, the rubric path (`${CLAUDE_PLUGIN_ROOT}/skills/managing-research-plans/references/plan-rubric.md`), the split-criteria path (`${CLAUDE_PLUGIN_ROOT}/skills/managing-research-plans/references/split-criteria.md`), the rubric version to stamp (`0.3`), today's date (`date +%Y-%m-%d`), and the step-1 resolution (component, version, the plan's `Provenance:` header, the master plan's `Initialized:` timestamp and the adoption-cutoff rule from step 1). The agent gathers its own git evidence and returns ONLY the scorecard JSON (the `board-scorecard` fence shape, schemaVersion 2). Validate it: parseable JSON, `threshold.verdict` ∈ pass/fail/undetermined, a grade block only on pass. Repair or re-prompt once on malformed output; still malformed → say so and fall back to running steps 3–4 inline. From the validated JSON apply the same reporting semantics as steps 3–4: **fail** → report `Threshold: FAIL — not a plan yet (…)` with the failures' near-miss language, fix list, and split assessment, offer the save, and STOP (no grade, no percent); **undetermined** → name the missing evidence, offer the save, and STOP; **pass** → report raw / applicable max, percentage, band, top three revisions, and the split assessment. Then continue at step 5.

3. **Stage 1 — Threshold (T1–T9; inline path — skipped when step 2 dispatched).** Run every check exactly as the rubric prescribes. Form is free: content that satisfies a check may live anywhere in the plan; when a template section is missing but the content is extractable elsewhere, the check passes and the missing section is named as an improvement. Apply T8's provenance-aware semantics: a *prospective* plan fails T8 only on undeclared retrospection (outputs predate an unlabeled plan); a *declared retrospective* plan passes T8 when its Sources resolve (step 1), lands `unknown` when they don't, and `fail` when they contradict the declaration. The verdict names the provenance class ("Provenance: prospective / retrospective") so a retrospective plan is never read as prospective.
   - **FAIL** (any check fails): report `Threshold: FAIL — not a plan yet (…)`, each failure with the rubric's near-miss verdict language, the nearest archetype (to-do list / prompt log / frozen preregistration / methods section — note that an *undeclared* reconstruction is a methods section, while a *declared* retrospective plan with resolvable Sources is admissible), a concrete fix list, and the split assessment. Offer the scorecard save (step 5). **STOP — do not grade, do not report a percent.**
   - **UNDETERMINED** (no fail, ≥1 unknown): name the missing evidence and how to supply it ("commit the plan to make prospectivity checkable"). Offer the save. **STOP — grade withheld.**
   - **PASS**: proceed to Stage 2.

4. **Stage 2 — Grade (G1–G8, only on PASS; inline path — skipped when step 2 dispatched).** Score each item with typed evidence: quoted plan text for text items; commit dates, version files, or log entries for G4. `N/A`/`unknown` items are excluded and shrink the applicable max by 2 each. Report raw / applicable max, percentage, band (below 50% revise before executing; 50–75% execute and address flags; above 75% strong), the top three concrete revisions (each actionable), and the mandatory split assessment per `${CLAUDE_PLUGIN_ROOT}/skills/managing-research-plans/references/split-criteria.md`. Judge the engagement gradient honestly: boilerplate that passed the threshold scores 0 at G1/G2/G6 — do not let a threshold pass inflate the grade.

5. **Offer to save the scorecard** to `plans/reviews/<NN-slug>-v<N>.md` using the template at `${CLAUDE_PLUGIN_ROOT}/skills/managing-research-plans/templates/review-scorecard.md` (schemaVersion 2: threshold block always present; on FAIL/UNDETERMINED `items: []`, null grade fields, band `"not a plan"`/`"undetermined"`; prose and JSON must agree — the board renders the JSON). Add a `"modelUsage"` field to the scorecard JSON: `{"prescribed": <the `plan-review` row from `models.py stage plan-review` as {"model","effort"}, or null>, "reported": {"model": "<the dispatched rp-plan-reviewer agent's pinned model, or the session model when you ran steps 3–4 inline>", "effort": <the pinned effort, or null inline>}}` — the board shows it as a "reviewed by" chip. If the file exists, ask before overwriting. Saved scorecards appear on the board's Reviews tab (`/research-plans:board`).

6. **If the researcher accepts a revision or a split, that is a decision** — append it to `plans/decision-log.md` (real timestamp), and route the change properly: plan revisions go through a new signed version (see `/research-plans:sync` step 6 semantics — never edit `vN.md`), splits go through tracker rows.
