# Design: v0.11 UI round (dark mode, soft-unwrap, provenance flow diagram)

Date: 2026-07-09. Status: reviewed. Three UI changes BK requested after merging v0.10 (PR #4). All changes are board-only (React app + one CSS file); no Python, command, or template changes except the version bump. Codex review (docs/specs/2026-07-09-codex-review-v0.11.md) folded in: BK approved all six corrections and the three defaults (diagram comments stay ordinary artifact comments with a recorded surface; the board live-follows OS theme changes when no override is stored; Label: lines never unwrap).

Decisions from the design interview: (a) dark mode follows the OS by default with a header toggle override, persisted in localStorage; (b) the linebreak fix is soft-unwrap — hard-wrapped paragraphs must flow to the container width, no mid-sentence breaks at source newlines (BK's own example: "The country-level variance is small once design weights are applied, and the pooled estimate is stable" as one flowing paragraph); (c) the provenance redesign is the script→artifact flow diagram with the existing ScriptViewer as the click-through reader.

## 1. Dark mode

Strategy: Tailwind v4 class variant. `board/src/index.css` gains `@custom-variant dark (&:where(.dark, .dark *));`. A `dark` class on `<html>` drives everything.

- Boot resolution (pure function `resolveTheme(stored: string | null, systemDark: boolean): "light" | "dark"` in `board/src/lib/theme.ts`, unit-tested): a stored `"light"`/`"dark"` wins; otherwise the OS preference. Applied **before first paint by a tiny inline script in `board/index.html`'s head** (vite-singlefile inlines it ahead of the app module — no flash of the wrong theme), with the same logic mirrored in `lib/theme.ts` for the React side. Storage key `rp-board:theme`, best-effort inside try/catch (file:// exports and the live server's per-port origins may not persist — the OS default then applies, which is acceptable). When no override is stored, a `matchMedia` change listener live-follows OS theme switches; an explicit toggle wins and stops following.
- Toggle: a reusable `ThemeToggle` component (☀/☾ button) rendered in the App header (right of the nav, left of the Feedback button), in the BatchGate header, and on the sent/approved/denied terminal screens — those all return before the main header, so each renders the toggle itself. Click flips the class on `<html>` and stores the choice.
- Sweep: every view and component gets `dark:` variants under a uniform mapping — page `bg-stone-50 → dark:bg-stone-950`; cards `bg-white → dark:bg-stone-900`; borders `stone-200/100 → dark:stone-800`; text `stone-900/800/700/600 → dark:stone-100/200/300/400`; muted `stone-500/400 → dark:stone-500` (unchanged where already mid-scale); hovers `hover:bg-stone-100 → dark:hover:bg-stone-800`; colored chips/badges/banners keep their hue with deep backgrounds (`green-50 → dark:green-950` etc., text lightened one step, borders darkened); the active nav pill `bg-stone-900 text-white → dark:bg-stone-100 dark:text-stone-900`.
- CSS file: `.prose-md` colors move to dark-aware rules (`.dark .prose-md { color: var(--color-stone-200); }` plus code/pre/blockquote/table/th/hr/a overrides); `mark[data-annotation]` gets a dark highlight (`.dark mark[data-annotation] { background: var(--color-amber-500/30 equivalent); color: inherit; }` — keep the mark readable, amber-300/25% opacity via color-mix); `.diff-add`/`.diff-del` get dark counterparts. `ScriptViewer` gets explicit dark selected-line and hover states (its `bg-amber-100` selection is light-only today).
- Files touched: `index.css`, `main.tsx`, `App.tsx`, all six views, `ArtifactCard`, `SafeTable`, `ScriptViewer`, `AnnotationLayer` (comment pill + textarea), `ReviewMenu`, `DiffView`, `Markdown` (class passthrough only), `BatchGate`, plus the new provenance component (§3). The exported/shared single-file boards inherit the toggle automatically (it is app JS).

## 2. Soft-unwrap (linebreaks)

Problem: the shared renderer uses `breaks: true`, so every source newline becomes a visible `<br>`. Hard-wrapped documents (agent-written reports, pasted sources, retrospective plans) render with mid-sentence breaks that fight the reader's screen width. `breaks: true` exists for a reason — line-oriented lines (`Serves: RQ1`, `Signed off: …`, `Stopping rule: …`) must stay on their own lines — so the fix is a preprocessor, not a flag flip.

- New pure function `unwrapSoftBreaks(md: string): string` in `board/src/lib/markdownText.ts`, applied in `Markdown.tsx` before `marked.parse`. Unit-tested contract:
  - Never touches fenced code blocks (``` … ```), and never joins table rows (`|…`), headings (`#…`), blockquotes (`>`), or horizontal rules.
  - Joins line N+1 onto line N (single space) when line N is non-empty prose (not a list marker line's own start, no trailing two-space explicit break) AND line N+1 starts with a lowercase letter or `(` — a sentence continuation — and is not an ordered-list line (`1.` / `1)`).
  - Everything else keeps its newline: `Label:` lines, `RQ1:` lines, new sentences starting uppercase, list items, blank lines.
  - BK's example round-trips: `…small once\ndesign weights are applied, and the pooled\nestimate is stable.` → one flowing paragraph. A wrap that falls on a sentence boundary (next line starts uppercase) keeps its break — a sentence starting on a new line reads fine; only mid-sentence raggedness is the bug.
- The provenance strip's `whitespace-pre-line` plan-goal rendering is replaced by the §3 header (which renders the goal through `Markdown`, so it gets the same unwrap).

## 3. Provenance flow diagram

Replaces the `<details>` "How these were produced" text list in `Results.tsx` with a Provenance section: collapsible, **open by default**, dark-mode aware, working identically in live and static/export modes.

- New pure builder `buildProvenanceGraph(bundle: ResultsBundle)` in `board/src/lib/provenance.ts` (unit-tested): takes the whole bundle (it needs `manifest`, `scripts`, and `assets`) and returns `{ scriptNodes, artifactNodes, edges }`. A script node per unique `producedBy.script` (label = basename, sublabel = `sourcePath`, lang, line count from the snapshot when present, `snapshotPath` = the snapshot's exact payload path for the ScriptViewer, or null when the snapshot is missing) plus one ghost node (`producer unknown`) when any artifact has `producedBy: null`. An artifact node per manifest artifact (id, title, kind, thumbnail url via `artifactDisplay` when the display mode is an image, `source.path` caption, tex/data chips). Edges: script → artifact. Snapshot resolution goes through a new shared `resolveScriptSnapshot(producedBy, scripts)` (suffix match, exactly what ArtifactCard does today at its line 24) exported from `lib/artifactDisplay.ts` and used by BOTH the card button and the diagram, so `openScript` always receives an exact `BoardFile.path`.
- New component `board/src/components/ProvenanceFlow.tsx`: two columns of HTML node cards (scripts left, artifacts right) over an absolutely-positioned SVG layer (`pointer-events: none`) drawing cubic-bezier edges between card anchor points — measured from refs after layout, re-measured via a `ResizeObserver` on the container plus thumbnail `onLoad` (annotation marks, font load, and details toggles all shift layout); no new dependencies.
  - Script node click → `setOpenScript(snapshotPath)` — the existing ScriptViewer drawer (line-numbered read, click-a-line comments) opens below, exactly the current UX. Missing snapshot → the node renders disabled with "snapshot missing".
  - Artifact node click → for images, the existing lightbox (`onZoom`); otherwise scroll to the artifact's card via a new `data-artifact-card-id="<id>"` attribute added to `ArtifactCard`'s root (never by querying the annotation scope, which would be ambiguous).
  - **Placement: the whole Provenance section renders INSIDE the same AnnotationLayer as the bundle body** (today's details block sits outside it, where selections silently would not route). Node text is real selectable text. Diagram nodes carry their own stamps — `data-annot-scope="provenance:<artifactId>"` (scripts: `provenance-script:<basename>`) — never a duplicate `artifact:<id>`. `Results.addSelectionComment` maps the `provenance:` prefix to an ordinary artifact comment (target kind `artifact`, same feedback shape — one comment type), and the annotation's target gains an optional `surfaceScope?: string` recording the original stamp so `paintable` repaints the highlight on the surface where it was made (diagram vs card) instead of the first scope that matches.
  - The section header keeps the capture line ("captured <date> under plan v<N>" with the retrofit phrasing) and renders the plan goal above the diagram via `Markdown` (one short block, replacing the old `whitespace-pre-line` paragraph).
- `Results.tsx` keeps a single `openScript` state feeding both the diagram and the per-card "produced by" buttons (unchanged).

## 4. Tests, build, release

- Unit tests (Vitest, no DOM): `theme.test.ts` (resolveTheme), `markdownText.test.ts` (unwrap contract incl. BK's example, fences, indented code, explicit two-space breaks, labels, lists, tables), `provenance.test.ts` (graph builder: grouping, ghost node, thumbnails, missing snapshots, `resolveScriptSnapshot` suffix matching). UI stays out of parse.test.ts per house rule.
- `npx tsc --noEmit`, `npm test`, `npm run build`; visual smoke in both themes via `npm run dev` against dev-data (which already has archives, validation, a typeset table).
- Version 0.11.0 in `plugin.json`, `marketplace.json`, `board/package.json`; CHANGELOG entry; README: one sentence for the dark-mode toggle, one for the provenance diagram (board section).

## Out of scope

- A three-column diagram with source-data nodes (source files are not embedded in the payload; revisit if a `/source/` live route is ever wanted).
- Theming the generated reports or PDFs (board only).
- Auto-layout for many-script pipelines beyond simple column stacking (rare today; nodes stack vertically and edges curve — fine into the tens).
