# Board control surface: docked feedback, always-on actions, persistent serving

Date: 2026-07-09
Status: sections A–D approved by BK; codex (gpt-5.6-sol, xhigh) review round folded in with BK's rulings on the two blockers
Target release: v0.15 (tentative — v0.14 is reserved by the models work in `2026-07-09-updates-sharing-models-design.md`; BK owns the release train)
Implementation timing: after v0.13 sub-plans 3–4 land (see §7)

## 1. Motivation

Three researcher complaints about the live board, from use at v0.11.0:

1. **Feedback is an overlay, not a workspace.** The feedback drawer is a fixed 320px panel pinned to the right edge (`App.tsx:739`). The main content is a centered 1024px column that never moves (`App.tsx:640`), so on any window narrower than ~1664px the drawer covers content. Drawer cards and their inline highlights have no linkage in either direction. BK: "feedback should be placed side by side."
2. **Researcher actions appear and disappear.** "Review with ▾" is hidden in Tracker and Plan views whenever the board opens in sign-off-gate mode (`!data.gate` guards at `Tracker.tsx:300`, `PlanReader.tsx:301`) but not in Results (`Results.tsx:344`) — the inconsistency BK noticed. (The Results-in-gate button is in fact silently broken: the server rejects `/api/feedback` whenever `gate_mode` is set, `board.py:598`.) The Tracker has no approve/pass action at all: plan approval exists only in the gate drawer (`App.tsx:866-888`) and the batch wizard (`BatchGate.tsx:196-224`), which are separate launch modes. Results verdict buttons vanish once a verdict exists (`Results.tsx:424`).
3. **The board dies under you.** The server is one-shot by design: the first successful action POST sets the done event and shuts the server down (`board.py:597-611`), the UI swaps to a "return to Claude" terminal screen (`App.tsx:545-563`), and an idle hour also kills it (`board.py:1144`, default `--timeout 3600`). `board.md:17` instructs a 10-minute foreground timeout for harnesses without background launches (the Claude Code Bash foreground ceiling). There is no keep-alive or reconnect anywhere in the client; later clicks surface "Could not reach the board server" (`App.tsx:857-865`). BK: buttons should always work, the whole interaction should be doable from the board, and the board should be reachable at a persistent link.

All line references are at commit `aa5ab97` (v0.11.0 on main). The feedback drawer opens by default in gate mode or when reviewer-seeded comments are present (`App.tsx:169-171`).

**Relationship to v0.13 (in flight).** v0.13 already answers the *collaborator* persistent-link ask: a password-gated Vercel-hosted snapshot that is comment-only, with researcher actions stripped at three layers. Nothing in v0.13 changes the local live board's lifetime. This spec is the *researcher-side* counterpart: it changes only the live board and must not weaken v0.13's channel rules (§5.3).

**Mode vocabulary.** Live = local server, researcher acting. Remote = single-file board a collaborator annotates and returns (comment + download only). Static = read-only snapshot, no annotation UI at all. Hosted (v0.13) = Vercel mirror, comment-only.

## 2. Design A — docked feedback panel

**Layout.** The header stays in the outer column; the content area below it becomes a flex row: `<main>` (flexible, `min-w-0`) plus the feedback panel as a real 380px column (`shrink-0`, left border) — the same sidebar pattern PlanReader/Results/Scorecard already use internally (`PlanReader.tsx:213`). Content reflows when the panel opens and is never covered. Panel closed → the column unmounts and content gets full width back. The panel is sticky and full-height with its own scrolling list and pinned footer, preserving the current internal structure (`App.tsx:751`, `:856`).

**Open triggers unchanged:** the header "Feedback (N)" button, adding any comment, gate mode, and reviewer-seeded comments (`App.tsx:169-171`).

**Two-way click-sync.** Highlights already carry their annotation id in the existing `data-annotation` attribute (marks created at `anchor.ts:217`); no new attribute is needed. Clicking a panel card navigates to the annotation via a typed navigation target: App lifts (or receives change handlers for) the child-view selections involved — PlanReader document index, Results bundle index and open script, Archive version index, Scorecard review index — switches tab/document as needed, clears or overrides Timeline filters that would hide the target, waits for render, then scrolls and flashes. A cross-element selection can produce several marks with one id: scroll to the first, flash all. Clicking a highlight opens the panel, scrolls to the card, and flashes it. Sync targets are keyboard-activatable and move focus, not pointer-only. Unanchored comments keep their pill (`App.tsx:775-779`) and show a "no highlight in this document" notice instead of scrolling. ScriptViewer gains stable line targets and receives saved script annotations (today it renders only the active selection, `ScriptViewer.tsx:63`).

**Narrow windows.** At and above a docking breakpoint the panel is a column; below it, today's overlay returns, gaining a scrim and click-outside-to-close (no scrim exists today). The breakpoint is chosen at implementation by a visual check at 1024px and 1280px with the Tracker's seven-column table and the PlanReader/Results internal sidebars in view — reflow must not become horizontal overflow.

**Draft persistence, stated precisely.** Saved feedback cards survive reloads and relaunches: the live-mode localStorage key becomes project-identity-based (today it embeds the payload hash, `App.tsx:126`, so a state-changing relaunch would orphan drafts), and a submission clears only the submitted annotation ids, not the whole key (`App.tsx:363` clears everything today). Open composer text and an un-submitted verdict reason are component state and are not promised to survive.

**Unchanged.** Drag-select → Comment pill → popover composer (AnnotationLayer) stays the one annotate gesture everywhere; no new comment buttons. GeneralCommentBox unchanged. Full dark-mode variants per the v0.11 sweep conventions.

## 3. Design B — always-available researcher actions

**Placement (live board only — `canPost`).** Action clusters on Tracker rows and the Plan view version strip: `[Approve vN] [Request changes] [Review with ▾]`. Results keeps `[Accept] [Request changes] [Review with ▾] [Generate report]` plus a Reopen affordance on decided bundles (below). Timeline, Reviews, and Archive are explicitly comment-only views — actions live where the acted-on object lives (Tracker/Plans/Results); `ReviewRequest.scope` stays `master | plan | results`.

**Gate interaction (BK ruling).** Review happens *before* the gate, not during it: with the persistent board, reviews can be requested at any time on the normal board, and the blocking gate stays a brief modal moment offering Approve / Request changes only. The `!data.gate` guards move from Tracker/PlanReader into a consistent rule: review controls are hidden in gate mode everywhere, which also removes Results' silently-broken in-gate review button (`/api/feedback` is rejected whenever `gate_mode`, `board.py:598` — unchanged). `--gate` and `--gate-batch` remain modal, synchronous launch modes with one-shot semantics.

**State-aware, not always-clickable.** A plan whose current version is already signed off shows a "✓ Signed off vN" badge instead of Approve; a plan with no draft pending sign-off (no sign-off line expected, missing file, pre-renewal/archived component) renders no approve control. Approve is disabled while pending comments scoped to that target exist (mirroring the single gate's rule, `App.tsx:875`); unrelated pending comments neither block nor ride along. "Request changes" carries the target-scoped pending comments plus a typed free-text reason field, required when no target comments are pending. (The batch wizard's note is optional today, `BatchGate.tsx:182` — the required-reason rule here is new, not a mirror.) Buttons disable while a submission is in flight.

**Sign-off authority — ticket, never the fence.** The sign-off hook mechanically guarantees that approval artifacts come only from a researcher-facing board subprocess: batch tickets are accepted only because agents cannot write them (`signoff_gate.py:228` denies agent-written tickets as forgery), and the blocking gate authorizes exactly one pending write via its exit code (`signoff_gate.py:325`+). A fence-carried "approve" order executed by the session would be Claude writing a signed plan with no such artifact — forged approval. Therefore: when the researcher clicks Approve, the board server itself validates the typed request against the payload it is serving and writes a one-use approval ticket bound to `{component, proposedVersion, path, normalized content hash of the exact displayed draft}` — the same trust position as batch tickets, generalized. The session then applies the sign-off write, which the hook admits by consuming the ticket. The feedback fence records the order for routing and audit but is never the permission proof. A stale draft (content hash no longer matching disk) → the POST is rejected with a refresh hint instead of issuing a ticket.

**Canonical document assembly for privileged actions.** Today the server writes the client-supplied `feedbackDocument` verbatim (locked in by `test_board.py:448`). For action-carrying submissions, `board.py` validates a typed request body against the served payload and assembles the canonical feedback document itself; the client's rendered markdown is advisory only.

**Verdict and Reopen (BK ruling: reopen = revision request).** Verdicts stay write-once (`results.py` creates `verdict.json` with `O_EXCL`; status spelling `accepted | changes-requested`, `results.py:9`). Accept/Request-changes on an undecided bundle ride the existing `## VERDICT:` order. On a decided bundle, Reopen files a change-request work-order against rN carrying the researcher's comments/reason: `verdict.json` is untouched, the rN badge remains a historical fact, the routed request drives the next capture (rN+1, with its own verdict), and the reopened intent is visible through the existing revision-request and decision-log surfaces — no new verdict state.

**Work-order channel.** All actions ride `POST /api/feedback` work-orders as today (`feedback.ts:64-98`): the new `## SIGNOFF:` order (approve | request-changes) plus the existing VERDICT / REVIEW REQUEST / REPORT REQUEST orders. The machine-readable `json board-feedback` fence gains the `signoff` key. `parse_fence` accepts arbitrary JSON (`board.py:958`) so parsing needs no schema change, but `feedback.ts`, `inspect_feedback_document` routing, and the fence tests on both sides are extended together. No new public endpoint semantics beyond the single-slot rule in §4.

## 4. Design C — persistent board: relaunch loop on a stable port

**Stable, pinned port.** `/board` derives a per-project default port — `41000 + (sha256(canonical project root) % 1000)`, probing upward if busy — and then **pins** whatever port it actually bound (recorded in the loop's state for the session): every relaunch reuses the pinned port with bind-retry (`server_close()` on shutdown, brief retry while the prior socket closes) rather than re-probing, so the loaded page's origin stays valid. Same project → same `http://127.0.0.1:<port>` bookmark across sessions while the canonical root path is unchanged; `--port` overrides. Cross-project collisions fall to probing, sacrificing bookmark stability for the second project — accepted.

**Loop protocol (board.md step 4–6 rewrite).** The one-shot server core is unchanged; persistence lives in the command loop:

1. Launch `board.py` serving in the background on the pinned port. The first launch may open a browser tab; relaunches pass `--no-open`.
2. Researcher clicks an action → the server validates it, persists the work-order durably (unique action id, atomic write; the order file is kept until its work completes and is only then acknowledged — a crash re-offers the unacknowledged order on the next launch, aligning with the existing `--collect` recovery), issues any approval ticket (§3), replies `200 {ok, actionId, bootId, projectId}`, and exits 0. Later action POSTs to the same server instance get `409` (single slot per round; the batch wizard keeps its multi-accept endpoints).
3. The session is re-invoked, executes the work-order (route comments, apply the ticketed sign-off, run the requested reviewer or report — reviewer runs can take many minutes), acknowledges the order, relaunches the board on the pinned port with a payload regenerated from disk, and ends its turn; repeat.
4. Exit code 2 (idle hour) or SIGTERM → the loop ends, no relaunch. Gate and batch launches do not loop; a gate round runs between loop iterations under the existing `.board.lock` arbitration (the loop server is down while a gate is up, and `/board` invoked while a live loop already owns the lock reports the running board instead of double-serving).

**Client reconnect — a small tested reducer.** States: `online → submitting → accepted → applying → online′`, plus `sleeping` and `stalled`. `/api/health` (`board.py:569`, currently unused by the UI) returns `{bootId, generation, projectId}`; HTML and health responses carry `Cache-Control: no-store`. The client's baseline `bootId` comes from the accepted POST response — never from a pre-submission poll (a poll answered by the replacement server would make the client wait forever). While `applying`, poll ~3s until a server answers with the **same projectId and a different bootId**, then reload (a different project on the same port must not trigger reload). Reload keying on bootId rather than generation matters: an action that changes no payload bytes never produces a new generation. After an accepted order, the UI never offers the copy-as-markdown fallback for that order (double-execution risk); a long delay shows "still applying — reviewer runs can take many minutes" plus a `/board` recovery instruction. The copy fallback remains only for POSTs that were never accepted. While `online` and idle, several consecutive failed polls (not one) move the UI to `sleeping`: banner "board sleeping — run /board to wake", action buttons disabled — no dead clicks.

**Sleep rules.** The board lives as long as the session loop runs; an idle hour puts it to sleep; the bookmark revives on the next `/board`. Every relaunch re-reads `plans/`, so the board also picks up work done outside it.

## 5. Security and trust boundaries

1. **Approval authority is artifact-based, never fence-based** (§3): one-use, content-hash-bound tickets written only by the board server subprocess; the hook's existing agent-write forgery denial extends to the generalized ticket.
2. **Privileged requests are validated server-side** against the served payload; canonical document assembly in Python (§3).
3. **Channel rules.** `signoff` (and every researcher-action key) is honored only from live-server ingress. `--pull` (hosted) and `--collect FILE` (remote/hand-delivered) route comments only, regardless of any keys a fence claims — a collaborator hand-editing a remote file's fence to add `signoff`/`verdict`/review keys gets those keys stripped at routing, extending v0.13's hosted-side stripping to every non-live ingress path. `signoff` is added to v0.13's action-key strip lists and hosted golden fixture. `canPost = mode === "live"` stays the UI gate after the hosted mode lands.
4. **Local server hardening matters more with a deterministic long-lived port**: keep v0.13's `local_request_ok` origin/host guard on mutating routes, add a per-boot token required on every mutating route, apply Host validation to GET as well as POST (DNS rebinding), and send frame-denial headers (clickjacking of privileged buttons). (Inference from the current GET handler serving HTML without Host or frame checks, `board.py:569`.)
5. **Verdict immutability is preserved** — no overwriting `verdict.json`, no `RESEARCH_PLANS_NO_GATE` shortcuts; Reopen never mutates the record (§3).

## 6. What this does NOT do

- No researcher actions on hosted/remote/static boards; static stays read-only, remote stays comment+download.
- No detached daemon mode (board surviving with no session attending); the loop design leaves room to layer it on later.
- No web-template changes; no change to the one-shot request core or the gate/batch protocols; no verdict event model.
- No mid-gate review protocol (BK ruling; revisit only if review-before-gate proves insufficient in practice).

## 8. Sequencing against v0.13 (why build-after)

v0.13 sub-plan 3 (hosted React UI) rewrites the exact seams this spec touches — the mode/capability flags (`App.tsx:122-125`), the drawer footer, `types.ts` mode union — and sub-plan 4 rewrites `commands/board.md` routing and `board.py` argument dispatch. Building now would mean merging against a moving branch and re-doing their work by hand.

Decision (BK, 2026-07-09): finish this spec and its codex review now; write the implementation plan and build in a worktree branched from `v0.13-sharing` after sub-plans 3–4 land, binding to their final code. Truly independent pieces may ship earlier if they turn out clean. Coordination note to hand the v0.13 execution: add `signoff` to the action-key strip lists and hosted golden fixture (§5.3) — cheap for them to include, and this spec's §5.3 restates it as a requirement either way.
