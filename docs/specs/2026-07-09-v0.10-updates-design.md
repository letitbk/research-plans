# Design: v0.10 updates (journal outputs, /renew, report generation, validation)

Date: 2026-07-09. Status: reviewed. Covers four features BK requested, designed together because they share surfaces (results bundles, the board, CLAUDE.md conventions). Codex review (docs/specs/2026-07-09-codex-review-v0.10.md) is folded in: BK approved all eight grounded corrections, the data + tex table schema, the pure-helper test approach for the artifact card, and the five defaults for Codex's open questions.

Decisions BK made in the design interview:

1. Journal outputs: full pipeline (init interview, CLAUDE.md conventions, execution guidance, capture preferences, board rendering).
2. /renew: archive the old master plan and write a fresh one. Numbering continues across renewals. Prior components the researcher still relies on are carried into the new tracker. The board gets a dedicated Archive view.
3. Report generation: per-bundle report, button on the Results view plus a `/research-plans:report` command. Formats: markdown always, PDF and DOCX via pandoc.
4. Validation: automatic at every capture, run by an independent subagent, sealed inside the bundle, advisory (never blocks capture).

## 1. Journal-quality outputs

Problem: agents finish analysis components with CSVs and list them as evidence. The board then dumps CSV contents inline (any CSV under 200 KB becomes a full table via a naive converter in `SafeTable.tsx`). The researcher wants journal-ready figures and typeset tables as the deliverables, with CSVs demoted to click-to-open data.

### 1a. Init interview and CLAUDE.md conventions

`commands/init.md` step 3 gains one question: "Target journal for figures and tables?" Options: ASR, AJS, Social Forces, Demography, PNAS, Other (free text), None yet. "None yet" writes generic publication-quality defaults.

`templates/claude-md-section.md` gains rule 7. Init and renew substitute the journal answer:

```markdown
7. Output conventions — target journal: <journal>. Analysis deliverables are journal-ready, not raw output:
   - Figures: vector PDF plus a PNG preview, sized to a journal column, grayscale-safe. Use the /journal-figures skill if available; otherwise export to the same spec with standard tooling.
   - Tables: a typeset table (.png preview plus .tex source, booktabs style). Use the /journal-tables skill if available; otherwise modelsummary/kableExtra/gt to the same formats. A CSV of estimates is an intermediate, never the deliverable.
   - Every figure and table carries a title and a one-line caption suitable for the manuscript.
```

Init update mode (already re-writes the marked block) offers to add rule 7 to projects initialized before v0.10. When refreshing an existing block, init reads the current "target journal:" value and re-asks only if it is absent.

### 1b. Execution guidance

`templates/execution-plan.md`: the Verification section comment gains one line telling the author to name the figure and table files the component will produce. `commands/plan.md`: the scope interview's "outputs" dimension points at CLAUDE.md rule 7. `skills/managing-research-plans/SKILL.md`: one convention line, analysis components deliver journal-ready figures and tables per the project's output conventions.

### 1c. Capture preferences (`commands/results.md`, `results.py`)

- Artifact preference order at capture: rendered figure (PDF/PNG) and typeset table (PNG plus .tex) first. When a finding's only support is a CSV, the capture step offers to render it now (via the skills or the fallback tooling) before staging, in line with rule 7.
- Table artifacts carry the typeset render as the thing the board displays: `file` points at the .png render, and on the Results view the table appears inline like a figure (findingMode embeds and the gallery). That display is the goal. Two optional fields attach the sources as quiet footer links, not co-equal artifacts: `tex` (new) for the .tex source and `data` (already in the schema, now actually consumed) for the CSV of estimates. Both are copied into `artifacts/` and hash-recorded, and `results.py validate_staged` checks them when present.
- CSVs and spreadsheets captured on their own (no render) get `kind: "other"`.
- `results.py`: add `.xlsx` to `SCAN_EXTS` so discover surfaces Excel files (captured as `other`, download card).

### 1d. Board rendering

- `board.py`: the inline whitelist shrinks to `.html` and `.md`. `.csv`, `.tsv`, `.tex`, `.json`, and `.txt` all stop inlining. (Today `tableKind` in `ArtifactCard.tsx` falls back to CSV rendering for any non-html/md inline text, so removing only .csv/.tsv would leave .tex mis-inlined.)
- The card's branch decision moves into a pure helper, `board/src/lib/artifactDisplay.ts`: `artifactDisplay(art, assets)` returns a display mode (`oversized | table-image | table-inline | figure | card | missing`) plus secondary links. `ArtifactCard.tsx` becomes a thin switch over the result. Unit tests in `artifactDisplay.test.ts` cover the contract: table with a .png renders as an image; a CSV never inlines, including legacy `inlineText` bundles; html/md still inline; tex and data links appear when the files are in assets. No new devDependencies; `parse.test.ts` stays parser/hash coverage only.
- `kind: "table"` with an image file renders as an `<img>` with the lightbox, same as figures, both in findingMode embeds and in the gallery. The card footer shows the `.tex` and `data` links when present, served by the existing `/artifact/` route in live mode and data URIs in export mode. CSV contents appear only when clicked.
- Legacy bundles whose table artifact is a bare CSV render as a file card (open/download) instead of an inline dump. This is the intended behavior change; note it in the CHANGELOG.

## 2. /renew and the Archive view

Problem: adopt reconstructs what already happened; nothing supports "keep the accomplishments, change the direction." `templates/master-plan.md` line 10 sanctions editing the project frame "only when the project itself changes direction" but no command implements it.

### 2a. The renew command (`commands/renew.md`, new)

Positioning: renew is forward-looking (new direction over existing work). Adopt stays backward-looking (faithful reconstruction under the current direction). Renew is the preferred entry when initiating research-plans in a repo with existing work; adopt remains for full retrospective reconstruction.

Flow:

1. Preflight. Detect whether the project is initialized (both markers). Detect exploratory content (scripts, outputs, git history). Suggest committing current state before the archive move.
2. Evidence inventory (read-only). Same scan approach as adopt step 2: git log clusters, top-level dirs, script and output inventories, in-repo planning docs. Ask where out-of-repo docs live; never scan outside the repo unprompted. Output: a prior-work inventory (activities, reusable assets), lighter than adopt's activity digest.
3. Interview. New direction and target, new research questions (proposals inside AskUserQuestion options), data changes, constraints, target journal (rule 7). Confirm proposed new components with the researcher.
4. Carry-over (initialized repos). Per existing tracker row, ask carry over vs leave archived (multiSelect). Carried rows keep their number, slug, execution dir, status, and results; their Serves is re-mapped to the new RQs (or `—`).
5. Archive (initialized repos). `mkdir -p plans/archive`, move `plans/master-plan.md` to `plans/archive/master-plan-<YYYY-MM-DD>.md` (suffix `-2` on collision). The archived file is never edited again (convention plus hook, 2d).
6. Write the fresh `plans/master-plan.md`: marker line; new Project context; new RQ list; components table = carried rows plus new components. Numbering continues: next-available = max over current rows, execution dirs on disk, and archived trackers, plus one. `Initialized:` is copied from the archived plan (the honesty cutoff survives renewal). New line under it: `Renewed: <YYYY-MM-DD> — <reason>; supersedes archive/master-plan-<date>.md`. New section `## Foundations`: the renewal line, what was carried, what was not carried with a one-line reason each, pointers to reusable prior assets from the inventory.
7. CLAUDE.md: create or refresh the marked block (including rule 7 with the journal answer).
8. Decision log: append a renewal entry (what changed direction and why). On an uninitialized repo, create `plans/decision-log.md` and `plans/execution/.gitkeep` first (init duties).
9. Offers, not defaults: `/research-plans:adopt` for carried rows that lack execution plans (adopt's existing "tracker already enumerates components" path consumes them as-is); `/research-plans:results --adopt` for pre-existing figures worth bundling; `/research-plans:plan` for the first new component.
10. Verify artifacts on disk, suggest a commit (`plans: renew — <reason>`).

Uninitialized repo: same flow minus steps 4 and 5. `Initialized:` = now. Foundations holds the inventory of pre-init work. This is the "initiate in an existing exploratory repo with a new mindset" case.

### 2b. Template and command edits

- `commands/renew.md` frontmatter: full allowed-tools including `Write`, `Edit`, `AskUserQuestion`, and `Bash` scoped to `mv`, `mkdir`, `git`, `date`, `ls`, `python3` (the archive move is a `mv`; nothing else in the plugin needs it today).
- `templates/master-plan.md`: optional `Renewed:` line and optional `## Foundations` section, with comments; delete when unused (same pattern as Sequencing notes).
- `commands/init.md`: update mode gains "changing the project's direction → /renew"; the step-7 routing for repos with substantial prior work now presents both: renew (new direction, preferred when the researcher wants a fresh frame) and adopt (reconstruct as-is).
- `commands/adopt.md`: preamble gains the positioning sentence (adopt reconstructs under the current direction; for a pivot use /renew, which archives the plan and hands carried rows back to adopt). No functional change; adopt already supports hand-built tracker rows.
- `SKILL.md`: renewal conventions (archive immutability, Initialized preserved, numbering continues, Foundations).

### 2c. Board: Archive view plus light markers

Payload (`board.py` and `parse.ts`/`types.ts`, the duplicated-schema pair, update both):

- `collect_payload`: read `plans/archive/master-plan-*.md` into `payload.archives[]` `{ name, path, content, archivedOn }`. `payload_files()` and `allFiles()` both gain the archives (content hash and share hash then cover them). Focused `--share` omits archives, consistent with its existing omission of the decision log, reviews, and history; full export and full `--share` include them (a snapshot publishes your plans).
- Master-plan parsing gains `renewed { date, reason }` surfaced from the `Renewed:` line, defaulting to null for plans that have none (old plans stay valid, same tolerance as plans without RQs). Foundations stays part of the document body (Plan reader renders it).
- `types.ts`: new `ArchiveFile` type; `DocumentCommentTarget` gains an `archive` member carrying the archive path (today the union allows only tracker, timeline, and reviews).

Views:

- `App.tsx`: the fixed tab list gains Archive, shown only when `archives.length > 0`. The view renders each archived master plan with the existing master-plan parser: document body plus its components table; rows link to the Plan reader and Results where the execution dir still exists. Wrapped in AnnotationLayer (general and selection comments route with the archive target).
- Tracker header: a renewal line ("renewed <date> · archived plan") linking to the Archive view.
- Tracker orphan handling: today an execution dir with no tracker row renders as red Drift (`orphanGroups`). After renewal that would paint every archived component red. Fix: a group with no current-tracker row but a row in some archived tracker re-badges as a quiet "pre-renewal" chip; groups in neither stay red Drift. `collect_drift` in `board.py` likewise skips source-drift checks for archived-only components.
- Plan reader and Results: components present on disk but absent from the current tracker get the same "pre-renewal" badge (computed client-side). Still fully browsable, reviewable, annotatable.
- Reconcile (`/results` no-arg), `/sync`, and drift checks stay scoped to the current tracker. Archived work is never nagged about.

### 2d. Immutability hook

`signoff_gate.py`: new pure-file-policy branch. Write/Edit targeting an existing file under `plans/archive/` is denied; creation of a new archive file is allowed. Never opens a browser (same class as the results branch).

## 3. Report generation

### 3a. Generator (`commands/report.md`, new)

`/research-plans:report <component> [rN]` (default: latest bundle). Reads the master plan, the governing plan version, the bundle (manifest, report.md, validation, artifacts, verdict), and the decision log slice for the component. Writes `plans/reports/<NN-slug>-rN-report.md` with:

1. Header block: project, component, plan vN, bundle rN, verdict state, validation status, generated date.
2. Background and goal (from the plan's Goal and Context, rewritten as prose for a reader who has not seen the project).
3. Data and methods (from report.md's data-and-method note plus the scripts).
4. Findings (each metric: statement, value, note, status).
5. Figures and tables (embedded renders with captions; tables as PNG images, with a line noting .tex availability).
6. Validation summary (status plus the step table, from feature 4; "not validated" when absent).
7. Provenance appendix (scripts with paths and hashes, source drift state, related decisions).

Then pandoc: PDF and DOCX. Degrade honestly: markdown is always written; report which conversions succeeded and why any failed (pandoc or LaTeX engine missing). Figure references are written relative to `plans/reports/` (`../execution/<slug>/results/rN/artifacts/fig1.png`) so the markdown renders in editors and on GitHub, and pandoc runs from `plans/reports/` so PDF and DOCX embed the same files; nothing is copied. Reports are derived documents: regenerating overwrites, `plans/reports/` is committed (not gitignored), and nothing about them is immutable. `commands/report.md` frontmatter: `Read`, `Write`, `AskUserQuestion`, and `Bash` scoped to `pandoc`, `python3`, `ls`, `date`, `mkdir`.

### 3b. Board button

Mirrors the existing review-request wiring exactly:

- `types.ts`: `ReportRequest { component, resultsVersion }`, added to `FeedbackMeta`.
- `feedback.ts`: a `REPORT REQUEST` block in the feedback markdown plus the meta field.
- `App.tsx`: `requestReport` mirroring `requestReview` (POST `/api/feedback`).
- `Results.tsx`: a Generate report button next to Review with, gated on `canPost` (live mode only).
- `commands/board.md` step 5: a routing branch that detects `reportRequest`, runs the report flow, reports the output paths, and offers to reopen the board.

Constraint accepted: clicking the button ends the board session (the server terminates on first POST); the board reopens after generation, same UX as review requests.

## 4. Validation at capture

### 4a. Mechanism (`commands/results.md`)

New step between staging and finalize, for bundles whose provenance will be `planned` and where a signed plan exists. Precondition: `commands/results.md` frontmatter gains `Task` in allowed-tools (it has none today, so the subagent could not spawn; `board.md` already carries it for agent reviews).

- Spawn one independent subagent (fresh context; it did not do the work). Inputs: the signed plan `vN.md`, the staging dir (manifest draft, report.md, artifacts, scripts), the decision log, and a git log slice for the execution window.
- The subagent returns strict JSON: per-plan-step verdicts (`followed | amended | deviated-unrecorded | not-executed | unverifiable`, each with an evidence string), per-success-criterion verdicts (`met | not-met | partial | unverifiable`, each with evidence), and notes.
- The overall status is derived mechanically, not by the subagent's mood: all steps followed/amended and no criterion not-met → `conforms` (or `conforms-with-amendments` if any step is amended); any deviated-unrecorded, not-executed, or not-met → `deviations-found`; insufficient evidence → `unverifiable`.
- The capture agent writes `validation.md` (readable audit: status header, step table, criteria table, evidence) and `manifest.validation` into staging, then finalizes as usual.

Malformed validator output: one repair retry (the same validate-and-reprompt pattern the board's agent reviews use), then record `{ status: "unverifiable", reason: "invalid validator output" }` and proceed. Never blocks capture.

Skips, recorded honestly in the manifest instead of silently: retrofit bundles get `{ status: "not-applicable", reason: "retrofit" }` (no governing prospective plan to validate against); headless runs without the Agent tool, or an explicit researcher opt-out, get `{ status: "skipped", reason }`. No `validation.md` in those cases.

Advisory: finalize proceeds regardless of the verdict. Deviations found are echoed to the researcher during capture; the remedy for an unrecorded material deviation is the existing one (`/sync` step 6, a new plan version), and `sync.md` gains one sentence linking validation findings to that step.

### 4b. Schema

`manifest.validation` (optional field; manifest `schemaVersion` stays 1):

```json
{
  "status": "conforms | conforms-with-amendments | deviations-found | unverifiable | not-applicable | skipped",
  "validatedAt": "YYYY-MM-DD HH:MM",
  "planVersion": 2,
  "validator": "subagent",
  "steps": [{ "planStep": "…", "verdict": "followed", "evidence": "…" }],
  "criteria": [{ "criterion": "…", "verdict": "met", "evidence": "…" }],
  "notes": "…"
}
```

`results.py validate_staged`: when `manifest.validation` is present, check the enums and shape; require `validation.md` only when status is a real verdict (`conforms*`, `deviations-found`, `unverifiable`). Absent field = valid (old bundles).

Board contract: the board reads `manifest.validation` only. `validation.md` is for humans reading the repo and for the report generator; it rides the existing payload hash via `manifest.json` (`manifestRaw` already covers the validation block) and is deliberately not added to `payload_files()`/`allFiles()`.

### 4c. Board

- `types.ts`: `ValidationBlock` on the bundle manifest type.
- `Results.tsx`: a Validation section (collapsible) between the report and the findings: status chip (conforms green, with-amendments blue, deviations amber, unverifiable gray, skipped/not-applicable muted), per-step rows with ✓/~/✗/○ marks and evidence, criteria rows. Scoped `data-annot-scope="validation"` so drag-select comments route.
- Tracker: the results badge on a component gains an amber mark when its latest bundle has `deviations-found` (computed client-side from the manifest, no `board.py` change).

## 5. Cross-cutting

- Hash contract: `payload_files` (Python) and `allFiles` (TypeScript) change in the same commit (archives added), with a shared fixture exercised by both `tests/test_board.py` and `parse.test.ts` (both already enumerate the hashed file set).
- Contract tests in `board/src/lib/parse.test.ts` cover: Renewed and Foundations lines in the master plan (renewed defaults null for old plans), the archives payload in `allFiles`, and `manifest.validation` parsing with the field absent (old bundles). Card behavior lives in `artifactDisplay.test.ts` (1d), not in parse tests.
- Python tests: `.xlsx` discovery, validation shape checks and tex/data checks in `validate_staged`, the archive branch in `signoff_gate.py` (placed as an early pure file-policy branch alongside the results branch, before the version-gate regex).
- `Results.tsx` guards every new field (`manifest.validation`, `tex`, `data`, `renewed`, `archives`) as optional; pre-v0.10 payloads render unchanged.
- `npm run build` regenerates the committed board template.
- Version 0.10.0: bump `plugin.json`, `.claude-plugin/marketplace.json`, and `board/package.json` (all three are at 0.9.2), CHANGELOG entries per feature.
- Docs: README (commands table gains renew and report, board section gains Archive view, Results section gains validation), QUICKSTART mentions renew for existing repos.

## 6. Out of scope (this release)

- Project-level reports (the per-bundle generator is written so a project report can later concatenate per-component sections).
- Rendering generated reports on the board.
- Master-plan versioning or board diffs between archived and current master plans.
- Validation as a gate (it stays advisory; verdicts remain recorded acts).
- Bundling figure/table renderer scripts into the plugin (conventions name the skills with a fallback instead).

## 7. Defaults chosen (flag in review if wrong)

1. `Initialized:` is preserved across renewal; `Renewed:` is a separate line. The honesty cutoff never moves.
2. One continuous decision log across renewals; the renewal itself is an entry, and pre-renewal entries stay valid.
3. Legacy CSV table artifacts stop rendering inline after the board upgrade (file card instead). No data migration.
4. `plans/reports/` is committed; reports regenerate and overwrite.
5. One validator subagent per capture (no panel), default model, advisory.
6. Archived master plans are immutable by convention and by hook (creation allowed, edits denied). No git hook; the honesty scope matches vN.md and rN today (shell moves stay convention-controlled).
7. Renew confirms with the researcher immediately before moving `master-plan.md` to the archive, and suggests committing first.
8. Focused `--share` omits archives; full export and full `--share` include them.
9. The board contract for validation is `manifest.validation` only; `validation.md` stays out of the payload.
10. Report figure paths are relative to `plans/reports/`, and pandoc runs from there; no file copying.
11. The target journal is canonical in CLAUDE.md rule 7 only. Init and renew read and preserve it on refresh; the report generator reads it from there. No second copy in the master plan.
